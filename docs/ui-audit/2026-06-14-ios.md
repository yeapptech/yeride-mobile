## Findings by Screen

### AddPaymentMethodScreen ❌

- **[critical] CardForm with no KeyboardAvoidingView — keyboard covers the Save button on small screens** — AddPaymentMethodScreen renders Stripe's `<CardForm>` (height: 280) followed by an optional error banner, then a Save button pinned at the bottom via `<View className="px-6 pb-4">`. There is no KeyboardAvoidingView wrapping the layout. On iPhone SE (375x667pt) the numeric/card keyboard is approximately 260pt tall. The card form + subtitle text already consume ~330pt, leaving ~337pt below the notch. When the keyboard appears, the Save CTA and the error banner are pushed off-screen with no scrollable escape, making the button unreachable without dismissing the keyboard first.  Fix: Wrap the root SafeAreaView content in a `<KeyboardAvoidingView behavior="padding" style={{ flex: 1 }}>` (iOS-correct behavior). Alternatively, place the CardForm and error banner inside a ScrollView with `keyboardShouldPersistTaps="handled"` so the rider can scroll to the Save button while the keyboard is up.
- **[warning] Save button bottom padding does not account for home indicator** — The Save CTA is in `<View className="px-6 pb-4">` which provides 16pt of bottom padding. The SafeAreaView wraps the whole screen but `edges` is not specified (defaults to all edges on react-native-safe-area-context), so the safe area inset should be applied — however the padding is on an inner View, not on the SafeAreaView itself, so the inset and the pb-4 may stack unpredictably. More critically the SafeAreaView without explicit `edges` prop may not apply bottom insets consistently across all iOS versions. Verifying with explicit `edges={['top','bottom']}` is the safe pattern used elsewhere in this codebase.  Fix: Add `edges={['top', 'bottom']}` to the SafeAreaView, or use `useSafeAreaInsets()` and apply `paddingBottom: insets.bottom + 16` to the bottom View containing the Save button.

### ForgotPasswordScreen ⚠

- **[warning] No ScrollView — keyboard covers submit button on small screens** — The form content sits inside a plain `View className="flex-1 justify-center"` with no ScrollView and no `keyboardShouldPersistTaps`. When the iOS keyboard rises, `KeyboardAvoidingView behavior="padding"` shrinks the available height but there is nowhere for the overflowing content to scroll. On iPhone SE (375×667 pt) the 'Send reset link' button and 'Back to sign in' link are pushed below the visible area and become unreachable without dismissing the keyboard first.  Fix: Replace the inner `<View className="flex-1 justify-center">` with a `<ScrollView contentContainerClassName="flex-grow justify-center" keyboardShouldPersistTaps="handled">` so content can scroll above the keyboard. This matches the pattern used by LogInScreen and RegisterScreen.
- **[info] Missing keyboardShouldPersistTaps — tapping outside TextInput does not dismiss keyboard** — There is no ScrollView with `keyboardShouldPersistTaps="handled"`. Tapping on the Pressable buttons or anywhere outside the email TextInput while the keyboard is visible will dismiss the keyboard instead of firing the press handler, requiring a second tap to submit.  Fix: Wrap content in a ScrollView with `keyboardShouldPersistTaps="handled"` (same fix as the keyboard-covers-button issue above).
- **[info] Hardcoded color `#000` on ActivityIndicator** — `<ActivityIndicator size="small" color="#000" />` at line 80 uses a raw hex value instead of a NativeWind theme token. On a dark-mode or themed background the spinner will be invisible against a dark primary button.  Fix: Replace `color="#000"` with a theme-derived value consistent with the button's foreground color token.

### RouteSearchScreen ⚠

- **[warning] Hardcoded color values in autocompleteStyles object** — autocompleteStyles at the bottom of the file uses raw hex values: `borderColor: '#e5e5e5'` (used twice, for textInput and listView) and `backgroundColor: 'white'` (used twice, for textInput and listView), and `placeholderTextColor: '#9ca3af'` in the textInputProps of both GooglePlacesAutocomplete widgets. These bypass the NativeWind theme token system and will not respond to dark-mode or theme changes.  Fix: Replace '#e5e5e5' with the --border CSS variable resolved at runtime (e.g. via useThemeColor or a shared constant), replace 'white' with --card or --background, and replace '#9ca3af' with --muted-foreground. Since Tailwind classes cannot reach inside the widget's style prop, use the app's theme color constants rather than hardcoded hex.
- **[warning] Bottom Continue button has no bottom safe-area inset** — The Continue button is in a plain `<View className="border-t border-border px-4 py-3">` at the bottom of the screen. RouteSearchScreen uses `<SafeAreaView edges={['top']}>` so only the top edge is inset. On iPhone models with a home indicator (iPhone X and later), the button sits flush against the bottom of the screen with only `py-3` padding, which may be partially obscured behind the home indicator bar.  Fix: Either change the SafeAreaView to include `edges={['top', 'bottom']}` and move the button inside it, or wrap the bottom button View with `useSafeAreaInsets()` and apply `paddingBottom: insets.bottom + 12` (or similar) so it clears the home indicator.

### RideReceiptScreen ⚠

The outer SafeAreaView uses `edges={['top']}` only, so the bottom of this fixed bar is not inset from the home indicator. On iPhone X+ the Done button will be partially hidden behind the system home indicator bar. ### RideScheduledConfirmationScreen ⚠

- **[warning] Bottom Got it button relies only on pb-6 with no safe-area inset** — The Got it Pressable sits inside `<View className="px-6 pb-6">` at the bottom of the screen. The SafeAreaView has no explicit `edges` prop, which defaults to all edges — this may work, but the bottom inset is consumed by the SafeAreaView while `pb-6` (24pt) adds extra space on top of that. However if the safe-area inset is not applied correctly (e.g. inside a modal stack), the button can end up behind the home indicator. The codebase pattern (see RouteSelectScreen, ActivityScreen) always uses explicit `edges` to be deterministic.  Fix: Add `edges={['top', 'bottom']}` explicitly to the SafeAreaView. This matches the codebase pattern and ensures the bottom button clears the home indicator on all iPhone models.

### DriverNavigationScreen ⚠

- **[warning] End Navigation button uses a hardcoded safe-area bottom inset instead of useSafeAreaInsets()** — The absolutely-positioned container for the End Navigation Pressable applies `style={{ paddingBottom: Platform.OS === 'ios' ? 28 : 16 }}` (line 164). 28pt is the home-indicator height on iPhone X-class devices, but it does not adapt to iPads (no home indicator, safe area is 0 or 20pt from status bar), iPhones with a non-standard inset, or landscape orientation where the inset changes. On devices where the actual safe-area bottom inset differs from 28pt the button either overlaps the home indicator or wastes space.  Fix: Replace the hardcoded paddingBottom with the live inset from `useSafeAreaInsets()`: `const { bottom } = useSafeAreaInsets();` then `style={{ paddingBottom: bottom || 16 }}`. Remove the `Platform.OS` branch — `useSafeAreaInsets` already returns 0 on Android.

### DriverEarningsScreen ✓

### TripDetailScreen ⚠

- **[warning] Bottom safe-area inset not applied — content scrolls under tab bar / home indicator** — All SafeAreaView instances use edges={['top']} only. The ScrollView's contentContainerStyle applies a fixed 16pt bottom padding, not a safe-area-inset-aware value. On iPhones with a home indicator (iPhone X+), and on both rider and driver stacks that render a bottom tab bar, the last Section ('Trip events') can be partially hidden behind the tab bar or home indicator when the list is scrolled to the bottom.  Fix: Add 'bottom' to the SafeAreaView edges prop on the success-state render path (line 162), OR keep edges={['top']} and pass contentInsetAdjustmentBehavior="automatic" to the ScrollView, OR use useSafeAreaInsets() and add insets.bottom to the ScrollView's contentContainerStyle bottom padding.
- **[info] Mixed spacing: hardcoded numeric padding in inline style alongside NativeWind className** — Line 167 uses contentContainerStyle={{ padding: 16 }} on the ScrollView while all child components use NativeWind className utilities (e.g. className="p-3", className="p-6", className="mb-4"). This mixes two spacing systems for the same visual concern on the same screen.  Fix: Replace the inline style with a NativeWind-compatible approach: either use className="p-4" on the ScrollView (if the NativeWind version supports it on ScrollView) or extract a token-based value via the theme — e.g. contentContainerStyle={{ padding: theme.spacing[4] }} — so spacing is consistent and derives from the same token system.
